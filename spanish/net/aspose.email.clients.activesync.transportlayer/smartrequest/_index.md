| [AccountId](../../aspose.email.clients.activesync.transportlayer/smartrequest/accountid) { get; set; } | Identifica la cuenta desde la que se envía un correo electrónico. Si AccountId es nulo, el servidor envía el correo electrónico utilizando la cuenta identificada por la configuración:PrimarySmtpAddress devuelta en la respuesta del comando Configuración. Si AccountId se incluye en la solicitud, el valor DEBE ser igual a uno de los valores del elemento settings:AccountId incluidos en la respuesta del comando Settings. El servidor DEBE validar que el valor del elemento AccountId proporcionado es válido para enviar correo electrónico. Se devuelve un valor de Estado de 166 si el valor del elemento AccountId no es válido. Se devuelve un valor de elemento de estado de 167 si la cuenta no admite el envío de correo electrónico. Nota El servidor envía el correo electrónico utilizando la cuenta especificada por AccountId, y no la cuenta especificada por From. El elemento AccountId no es compatible cuando la versión del protocolo es 12.1 o 14.0. Exchange 2007 devuelve el valor de estado 103 si el elemento AccountId se incluye en una solicitud de comando SendMail, una solicitud de comando SmartForward o una solicitud de comando SmartReply. |
| [ClientId](../../aspose.email.clients.activesync.transportlayer/smartrequest/clientid) { get; set; } | Especifica el ID de mensaje único (MID) del cliente. El valor de ClientId puede tener hasta 40 caracteres de longitud y DEBE ser único para cada mensaje, ya que el servidor usará el valor de ClientId para identificar mensajes duplicados. El valor de ClientId puede ser un simple contador incrementado para cada nuevo mensaje. |
| [Mime](../../aspose.email.clients.activesync.transportlayer/smartrequest/mime) { get; set; } | Contiene el mensaje codificado en MIME. El contenido Mime se transfiere como un BLOB opaco dentro de las etiquetas WBXML, como se especifica en [WBXML1.2]. Si el mensaje contiene una solicitud de reunión, el elemento Mime contiene los detalles de la reunión en formato iCalendar [MS-OXCICAL] o formato de encapsulación de transporte neutral (TNEF) [MS-OXTNEF]. Como se especifica en [RFC2447] sección 3.4, las solicitudes de reunión de iCalendar tienen un tipo de contenido de "texto/calendario" con el parámetro de método establecido en "SOLICITUD". |
| [ReplaceMime](../../aspose.email.clients.activesync.transportlayer/smartrequest/replacemime) { get; set; } | Especifica si el cliente está enviando el mensaje completo. Cuando ReplaceMime es VERDADERO, el servidor NO DEBE incluir el cuerpo o los archivos adjuntos del mensaje original que se está reenviando. Cuando ReplaceMime es FALSO, el cliente DEBE agregar el cuerpo del mensaje original como archivos adjuntos al mensaje saliente. El cliente puede usar esta propiedad para indicar si el mensaje se editó en línea o si el mensaje tenía texto de respuesta/reenvío antepuesto al mensaje de origen. Si ReplaceMime es VERDADERO, el mensaje fue editado. |
